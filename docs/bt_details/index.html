Behavior Tree - 进一步讨论

1.关于节点状态信息的讨论​

节点会向其父节点返回三种状态信息中的一个，这三种状态分别是：

  • 成功（Success）

  • 失败（Failure）

  • 运行中（Running）

  前两个，正如它们的名字，是用来向父节点通知运行的成功或失败的结果。第三种是指还在运行中，结果还未知，在下一次tick的时候再去检查这个节点的运行结果。这个功能非常重要，它可以让一个节点持续运行一段时间来维持某一行为。

例如，一个“行走(Walk)”节点会在计算寻路和让角色保持行走的过程中持续返回“Running”来让角色保持这一状态。如果寻路因为某些原因失败，或是因为发生某些状况行走的行为不得不中止，那么行走节点会返回“Failure”来告诉父节点；如果这个角色走到了目的地，那么行走节点返回“Success”来告诉父节点这个行走指令已成功完成。

为什么需要返回节点的运行状态呢？

  • 控制节点中，需要节点的运行状态来控制行为执行

  • 外部世界需要了解行为的运行状态，来决定是否更新决策

对于第二点，以“行走”为例说明:
当我们走向目的地的过程中，运行状态就是运行中；当行为完成或被中止，就返回成功或失败。这样，对外部来说，当我们看到节点在运行中的时候，就不需要做任何新的行为（为了优化，防止行为抖动）；当看到节点已完成时，就可以做新的行为决策了。

运行状态还有助于我们检测行为树的状态，帮助调试。

2.关于条件节点的讨论​

  条件节点用于判断某条件是否成立，以便对接下来的行为决策提供依据，其实就是要执行的行为的前提。那么为什么有时要将条件判断抽离出来作为一个节点单独处理，而不放在行为里面呢？这是为了贯彻职责单一原则，是一种更复杂也更成熟的做法。将其作为叶子结点放入行为树，辅助控制节点做决策，如此更加模块化，易复用。

以判断某目标是否在视野内为例，其实写在射击的行为中也可以，但这样的话这个条件判断和射击行为就不单一了，不利于节点的复用。

一般条件节点出现在序列节点中，其后紧跟条件成立后的行为节点。

3.关于序列节点的讨论​

序列节点按从左向右的顺序执行其子节点，并向其父节点返回状态信息：

  • 成功：所有子节点都返回成功时

  • 失败：当前子节点返回失败时

  • 运行中：当前子节点正在运行中

3.关于选择节点的讨论​

  我们说过选择节点是通过判断子节点的前提来选择一个节点执行，这就牵涉到判断顺序问题，是自左向右，还是随机选择，还是其他一些规则等等，这样就衍生出各种各样的选择节点。

3-1.带优先级的选择节点（Priority Selector）​

  这种选择节点每次都是自左向右依次检索，当找到一个可执行的子节点后就停止搜索后面的子节点。这样的选择方式，就存在优先级：最左边的子节点优先级最高，因为它是最先被判断的。对于带优先级的选择节点来说，它的子节点的前提设定，必须“从窄到宽”，否则后续节点就会发生“饿死”的情况，也就是说，后面的节点可能永远不会被执行到。
  为了更清楚地说明，请看下面图例。第一张图中，这三个子节点在一个带优先级的选择节点下，它们的前提会被依次判断，可以看到这个三个子节点的前提从左向右，一个比一个更严格。如果我们现在a=9，按照下图的定义会执行第一个子节点；如果a=7，则会执行第二个子节点；如果a=11，则会执行第三个子节点。下面的第二张图演示了一种节点“饿死（Starvation）”的情况，我们看到第一个子节点的前提，比第二个子节点更宽泛，只要a<10，那自左向右判断的话，只会进第一个节点。

所以，如果要用到带优先级的选择节点，则必须检查每一个子节点的前提，以防止节点饿死的情况。

3-2.不带优先级的选择节点（Non-priority Selector）​

  这种选择节点的选择顺序是从上一次执行过的子节点开始，如果前提满足，则继续执行此节点；如果条件不满足则从此节点开始，依次判断每个子节点的前提。当找到一个条件满足的子节点后，执行该节点。这种方式，是基于一种“持续性”的假设。因为一个行为一般不会一下子结束，而是会持续一段时间，所以有时为了优化，我们可以优先判断上一个执行的节点，当其条件不满足时，再寻找下一个可执行的节点。这种寻找方式不存在哪个节点优先判断的问题，所以对于前提的设置要求，就是保证“互斥”。如果我们用上面第一张图来说明，把控制节点换成不带优先级的选择节点，可以看到，当a=3时，第二个子节点会被执行，下一次当a=9时，由于不是从头依次判断前提的，所以，我们还是会选择第二个节点，而不是我们更期望的第一个节点。正确的做法见下图，注意每一个子节点的前提是互斥的。

  所以对于不带优先级的选择节点，它子节点的排列顺序就不是那么重要了，可以任意排列。

3-3.带权值的选择节点（Weighted Selector）​

  对于这种选择节点，我们会预先为每一个分支标注一个“权值（Weight Value）”，然后当我们选择的时候，采用随机选择的方式来选。随机选择会参考权值，并且保证已经被测试过的节点的不会再被测试，直到有一个节点的前提被满足，或者测试完所有的节点。由于随机，子节点的前提可以任意，而不会发生“饿死”的情况。一般来说，我们通常会把所有子节点的前提设为相同，以更好的表现出权值带来的概率上的效果。当所有子节点的权值一样时，选择节点就成为了随机选择节点（Random Selector），带权值的选择节点对于需要丰富AI行为的地方非常适用。比如养成类游戏中，小狗表示开心的时候，可能会有各种各样的表现，我们就可以用这种选择节点，添加各种子节点来实现。

  以上是三种常用的选择节点类型。当然，我们可以根据需要，定义更多不同的选择节点来选择行为。我们可以看到，不同的选择节点对子节点前提的要求会有不同，这是我们搭建行为树时需要注意的地方。

4.关于并行节点的讨论​

  对于控制节点来说，它的运行状态就是其子节点的运行状态。选择节点和序列节点比较好处理，因为这两种控制节点每时刻只有一个子节点在运行。但对于并行节点来说，它同时刻会有多个子节点运行，我们如何来处理并行节点的运行状态呢？一般有两种：

  • 与：只有所有子节点都运行完成，才返回完成
  • 或：只要有一个子节点运行完成，就返回完成

  并行节点通常用来处理并发行为。执行两个子节点，可以理解为，如果执行A任务，那么B任务也会执行，A是主要任务，B是次要任务。如果主要任务失败，则返回失败；如果主要任务成功，则返回成功。根据需要设定，A完成后，可以等待直到B执行结束，也可以直接向父节点返回结果，并停止执行B。

例如角色正在被丧尸追赶，需要同时执行逃跑和观察四周有无丧尸。若逃跑失败，则失败。若逃跑成功，已进入安全区，可以继续观察四周有无丧尸，也可以直接返回成功，停止观察，进行新的行为决策，如躺下休息。

5.关于装饰器节点的讨论​

  装饰器节点，顾名思义，就是用来装饰辅助的节点，可以自定义修改其子节点返回的结果。装饰器节点只能有一个子节点。

  • 中断器（Interrupt）：中断器被分配了条件判断，如果条件成立，则其子节点运行被中断，中断器返回失败；否则，中断器执行其子节点并返回子节点返回的状态信息。
    • 成功：子节点返回成功时
    • 失败：子节点返回失败，或子节点正在运行时中断器的条件为true或变为true
    • 运行中：子节点返回运行中

  • 取反器（Inverter）：取反器会将其子节点返回的成功或失败重新映射为相反的状态。

    • 成功：子节点返回失败
    • 失败：子节点返回成功
    • 运行中：子节点返回运行中

  • 中继器（Repeater）：中继器会重复其子节点N次，或直到子节点返回指定的状态，或永远重复

    • 成功：如果为“Repeat Times（重复次数）”，则在子节点最后一次重复后返回成功；如果为“Repeat Until（重复直到）”，则在指定状态为成功且子节点返回成功时返回成功
    • 失败：如果为“Repeat Times（重复次数）”，则在子节点最后一次重复后返回失败；如果为“Repeat Until（重复直到）”，则在指定状态为失败且子节点返回失败时返回失败
    • 运行中：只要重复就返回运行中

  • 暂停器（Timeout）：如果子节点的运行时间超过规定时间，则中断子节点运行；否则，返回子节点返回的状态信息。

    • 成功：子节点返回成功
    • 失败：子节点返回失败或超时未完成
    • 运行中：子节点返回运行中

    以上为四种常见的装饰器，根据需要，我们还可以自定义其他各种各样的装饰器。为了方便理解，应尽可能使用比较常见的装饰器。

装饰器应用举例
假如要在多个地点进行搜寻行为，如果地点数目较多，整个行为树可能会非常庞大，给我们进一步添加节点带来麻烦。此时我们可以引入装饰器来避免麻烦：相较于每增加一个搜寻地点就复制一份相同的子树，我们可以定义一个中继器（Repeater）来重复执行子节点，完成多个地点的搜寻。

6.关于具体实现的讨论​

  整个行为树就是不断地进行函数调用，它的树形结构就类似于图形化编程。我们需要定义行为树的输入与输出。一般来说，我们希望行为树是一个黑盒，也就是说，它仅依赖于预定义的输入。为了存储这些可以被多个节点共享的输入信息，我们可以抽象出一个容器，这里我们引入Blackboard的概念。黑板是一块可以被节点读写的公共存储区，将在之后介绍。
