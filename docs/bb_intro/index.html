Blackboard - 介绍

共享数据​

  在AI结构中，我们希望有一块区域，大家可以读取、存储一些信息，并且这块区域是全局共享的。共享数据的需求，从某种程度上说，是一种“偷懒”。试想一下，一块非常方便，存我想存的、读我想读的，非常容易理解的，非常容易架构（哈哈，或许吧，此时本人正在被黑板折磨）的结构，相信很多人都会把它作为第一选择。确实，共享数据优势很大，很有吸引力，在开发中，可以作为很多问题的解决方案。
  共享数据是一种缺点和优点同样明显的结构。首先，如果共享数据是被读取，那么只读操作不会影响到数据。但当一个或多个线程要修改共享数据时，我们谁也不知道谁在什么时候把数据修改了，如果数据不小心被莫名且错误修改，就会产生很多麻烦。所以，必须小心谨慎才能保证项目正常工作。另外，共享数据很容易乱。共享的数据随着开发进程不停地被修改，或添加新的变量，或删除冗余的变量。由于开发过程是多人协作的，很有可能存在两个作用完全相同的变量，或误删有用的变量，或项目需要的某些变量所有人都以为别人会写而导致最终项目缺少数据。如果没有得到好的维护和清理，共享数据会逐渐成为“垃圾堆”。另一方面，程序员容易过度依赖共享数据，认为任何东西都可以存在其中，而忽视对本身模块的架构。在某些情况下，这可能会成为阻碍解决问题的最后一根狗尾巴草。
  然而，共享数据究竟是黄金屋还是垃圾堆，不取决于共享数据本身，而取决于我们如何去实现、如何去维护、如何去规范、如何去使用。

黑板​

  Blackboard在人工智能领域已经是个古老的东西了，它来自一个直观的概念，就是一群人为了解决一个问题，在黑板前聚集，每个人都可以发表自己的意见，在黑板上写下自己的看法，也可以基于别人记录在黑板上的看法，来更新和发表自己的看法。在这样不断的意见交换、看法更新中，越来越趋于问题的最终解答。

基于上面的描述，我们可以看出黑板具有几个功能：

  • 读取：黑板上的内容谁都能自由读取
  • 记录：每个人可以写下自己的看法
  • 更新：调整已有的看法
  • 删除：删除过时的、错误的看法

  所以黑板本质上就是一个共享数据的结构。它对于多个系统间通信很有帮助。从程序设计的角度来说，它提供一种数据传递的方式，有助于系统的封装和解耦合。对于各个子系统而言，只需要把自己的运算的结果数据记录在黑板上，至于这个数据谁会去用，并不需要关心；反过来也是一样，对于自己的运算时需要用到的数据，可以从黑板上去获取，至于这个数据是谁提供的，也不需要关心。只要这个数据在黑板上，就认为是合法数据。这就提供了一种灵活性，各个子系统的设计也会相对独立。
当然黑板也有不足的地方，比如读写删操作相对随意，特别是写删操作，容易破坏数据，或者产生子系统间的竞争。比如，系统A和系统B都会去修改data1，那以谁的值为准呢？还有一个情况就是产生非法数据，一般认为，只要在黑板上的数据就是合法的数据，在读取时，不需要判断它是否合法，但如果一个子系统没有很好的维护它自己产生的数据（比如，该删除的时候没删除，或者赋值错误），那别人读取该数据之后，就会产生错误的运算结果。
所以，在黑板中，我们可以加一些功能：

  • 数据过期时间：对于写入黑板的数据，可以加一个过期时间，比如3秒后该数据过期。
  • 数据作用域：我们可以规定能够读写该数据的子系统。默认情况下，黑板的数据全局可见，就像程序中的全局变量一样，但如果我们希望某些数据只对个别子系统开放，可以通过作用域字段来指定。

既然我们选择使用黑板，那么它必然有自己的好处：

  • 解耦合：黑板作为独立的数据模块，超然于其他模块之外，提供数据维护和管理的功能。
  • 共享性：黑板的数据是共享的。我们想要获得一个数据，不必先知道提供它的实例，然后通过get方法去获取。我们只需要存一个黑板的实例，然后通过黑板获取数据的方法来获取数据。
  • 数据维护和管理：黑板提供数据的读写删操作、生命期、作用域等，让我们可以从管理数据的漩涡中解脱出来。

  有人可能会说，我把变量和传递方式一个一个具体定义，和存在黑板中好像差别不大，黑板带来的好处好像还不够多。但黑板还有一个优势，就是可视化和数据驱动。行为树经久不衰，一方面是因为它的概念直观，另一方面也是因为它在可视化和数据驱动方面的优势。黑板在这样的潮流中，一点也不落后。另外，黑板作为共享数据，可以很好地和行为树系统协同工作。我们将在下一节进一步讨论。
